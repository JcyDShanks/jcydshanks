Context

Java中的Context

上下文代表您的环境。它表示围绕系统所在状态的状态。例如，使用的运行环境，环境变量，实例变量，局部变量，其他类的状态，当前环境的状态等。

在某些API中，您会在接口/类中看到此名称，例如：

  • Servlet ServletContext
  • JSF FacesContext
  • Spring ApplicationContext
  • Android Context
  • JNDI InitialContext等

它们通常都遵循外观模式，该模式抽象了最终用户不需要了解的环境细节。

Android中的Context

应用程序环境的全局信息接口，是一个抽象类，其实现由Android系统提供。它可以访问特定应用程序的资源和类，以及对应用程序级操作（例如启动活动，广播和接收意图等）的调用。

它是应用程序/对象当前状态的上下文。它使新创建的对象了解正在发生的事情。通常，您调用它来获取有关程序另一部分（活动和程序包/应用程序）的信息。

Context代表环境数据，提供了对资源的访问。可以通过调用getApplicationContext()，getContext()，getBaseContext()或this获取上下文。

Context本身是一个抽象类，有两个实现类：ContextWrapper 和 ContextImpl 。

ContextWrapper

ContextWrapper是一个包装类，其构造器中必须包含一个真正的context，调用ContextWrapper的方法都会通过其所包含的真正的Context对象来处理。

ContextThemeWrapper在其内部包含了与Theme相关的接口，只有Activity才需要主题，所以Activity要单独继承自ContextThemeWrapper。

而Application、Service不需要界面，所以继承自ContextWrapper

ContextImpl

Context的具体实现类，实现了Context中所有的函数，应用程序调用的各种Context类的方法均来自于ContextImpl。

主要功能

  1. 加载资源
  2. 启动Activity
  3. 创建View
  4. 获取系统服务

作用域

在绝大多数场景下，Application、Service和Activity是可以通用的，但在一些特殊场景下比如启动Activity和弹Dialog，出于安全原因考虑，Android不允许Activity或Dialog凭空出现；一个Activity的启动必须要建立在另一个Activity的基础之上，也就是以此形成的返回栈。而Dialog则必须在一个Activity上面弹出（除非是System Alert类型的Dialog），因此在这种场景下，我们只能使用Activity类型的Context，否则将会出错。

Activity所持有的Context的作用域最广，因为Activity继承自ContextThemeWrapper，而Application和Service继承自ContextWrapper。ContextThemeWrapper在ContextWrapper的基础上又做了一些操作。

Application

由于Application继承自Context，包装类ContextWrapper中有一个attachBaseContext()方法，这个方法会将传入的一个Context参数赋值给mBase对象，之后mBase对象就有值了。

在mBase被赋值后，才可以调用Context的各种方法。

所以 Application的生命周期是

init() -> attachBaseContext() -> onCreate()

通常只有需要进行全局初始化的时候才自定义Application，另外 自定义Application不要用单例模式创建这个Application对象。Application对象全局只有一个，本身就是个单例，Application是属于系统组件，系统组件的实例是要由系统来去创建的，如果这里我们自己去new一个MyApplication的实例，它就只是一个普通的Java对象而已，而不具备任何Context的能力。

在Fragment中使用context

在Fragment中可以通过 getActivity()或getContext()两种方法获取context。

还有一种方法是在onAttach中保存context的引用。

或者可以在onCreateView()中，通过inflater.getContext()拿到context。

需要注意的是，在onAttach之前是无法获取到context的，无论调用以上那个方法都会返回为null。且如果在onAttach中存储context的引用，需要在onDetach中释放它。

当Fragment被Activity持有时， getActivity()或getContext() 没什么区别。

当不被持有时，getContext() 不一定是Activity。

最后说说外观模式。

外观模式

定义了一个高层、统一的接口，外部与通过这个统一的接口对子系统中的一群接口进行访问。

通过创建一个统一的外观类，用来包装子系统中一个 / 多个复杂的类，客户端可通过调用外观类的方法来调用内部子系统中所有方法。

主要作用
  • 降级复杂度，复杂用法封装。
  • 降低客户端与内部的耦合度。
  • 提高便捷性，客户端无需关心内部细节，直接使用。
